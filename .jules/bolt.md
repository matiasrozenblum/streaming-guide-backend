## 2025-03-05 - [N+1 DB Query Avoidance in Scraper]
**Learning:** Found N+1 query patterns inside the `insertSchedule` method of `ScraperService`. For every scraped item, it runs `await this.programRepo.find` and then for each day `await this.scheduleRepo.findOne`. This leads to poor performance.
**Action:** Always pre-fetch existing records into a Map or Dictionary outside of the loops to avoid N+1 DB calls.

## 2025-03-24 - [N+1 Redis Query Avoidance in Streamer Live Status]
**Learning:** Found N+1 Redis query patterns inside the `getLiveStatuses` method of `StreamerLiveStatusService`. For every requested streamer, it runs `await this.getLiveStatus(id)` concurrently wrapped in `Promise.all`, which executes individual `GET` commands to Redis. This leads to connection overhead and poor performance.
**Action:** Replace `Promise.all` with individual `get` calls with a single `mget` command to batch the retrieval, mapping the responses back to the input array indices.
## 2024-04-22 - [Performance Optimization] Parallelized Push Notifications
**Learning:** Sequential iteration over devices and push subscriptions causes unnecessary delays during bulk notification deliveries, especially if network calls block one another. Additionally, in `PushScheduler`, gating checks inside a loop evaluating against `ConfigService.canFetchLive` trigger O(N) network/database round-trips. Furthermore, completely replacing the repository `.gitignore` via shell redirection leads to massive unwanted diffs across unignored generated files (e.g., node_modules, coverage).
**Action:** Replace nested loops performing individual `await pushService.sendNotification` calls with concurrent execution using `Promise.allSettled()` on arrays of async tasks. For gating checks in a loop, pre-fetch configurations using a batched `Promise.all()` and cache the results in a `Map` for O(1) lookups. Never overwrite `.gitignore` with `>`; always use safe modifications or skip altogether if ignoring the node environment.
